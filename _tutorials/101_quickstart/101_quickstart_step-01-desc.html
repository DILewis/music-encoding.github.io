Welcome! To encode a single note, you'll need a <note/> element and the three attributes @pname (name of the pitch), @oct (octave) and @dur (duration).

So try to enter the following encoding: <note pname="c" oct="4" dur="4"/>.

Attention: This is also a tutorial on how to write tutorials now.

Tutorials consist at least of two main files:

  • a Markdown file (.md) that defines metadata (like a tutorial name and a corresponding JSON file) in the Jekyll front matter
  • and a JSON file (.json) in which all the tutorial steps are defined by the following options for each step:
    • label (string): the name for this step. If not specified, a default label will be created (Step 1, Step 2 etc.)
    • editorLines (integer): a specific height for the editor with integers (no strings please). If not specified, a default of 5 will be assumed
    • descFile (string): the name or path to a description file for this step
    • prefillFile (string): the name or path to a prefill file for this step
    • xmlFile (string): the name or path to a mei file for this step
    • xpaths (array): a set of xpath rules to be validated in this step, each one described by:
      • rule (xpath string): the actual xpath expression
      • renderanyway (boolean): a flag that, if set to true, tries to render the content, no matter if validation of the rule succeeds
      • hint (string): a hint message that will be displayed to the user as long validation does not succeed

As you can see above, each tutorial step has its own set of possibly three files defined in the JSON file. While the file names really don't matter (they just have to match a file in the folder of the tutorial), you maybe want to name them according to the corresponding tutorial step:

  • The descFile has an HTML description of the step instructions like those at the beginning of this page.
  • The xmlFile has the actual MEI encoding that is the final goal of the tutorial step. Two snippets can help to define different sections of the file:
    • An "edit snippet" can be inserted to mark the start (<?snippet-start?>) and end (<?snippet-end?>) of the section that the user is meant to edit or to enter into the editor. During user input, the preview box will be live updated automatically. For validation, the xmlFile's content between snippet start and end will be cut out and replaced by the user's input.
    • A "preview snippet" can be inserted to mark the start (<?preview-start?>) and end (<?preview-end?>) of the section that the user is meant to see in the preview box (if it is necessary to provide some additional context). The preview snippet has no influence on validation and it is expected to be placed outside of the preview snippet.
  • The prefillFile allows you to prefill the editor at any given step of a tutorial with the file's (XML) content, as the previous content of the editor will not be preserved for the next step. The content of prefillFile will be inserted into the editor, replacing it's content (if any). If done on the first step of a tutorial, this allows you to start with something in the editor right away (if not, the editor will start empty). Set at a later step, it allows to switch to a new "entry point" into the MEI file being edited. Like the comment in the editor below, the prefill will also appear in the preview box automatically.

Editor input is used during validation (see "edit snippet" above), so you may want to make sure that the edit section (prefill + user input) will end up in a MEI file structure ready for validation.
